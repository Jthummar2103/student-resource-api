Description

In this phase of the Course Project, you will architect and build the "Headless" Backend Engine that will power your final application. Working individually, you will create a RESTful API using Node.js, Express, and a relational database (MySQL/MariaDB).

This API will serve as the foundation for your Front-End application in Phase 2. Your goal is to build a secure, robust, and well-documented service that handles data persistence, user authentication, and business logic. There is no UI/Frontend component for this deliverable.

Rationale

This assignment will evaluate the following Course Learning Outcomes:

  • CLO1: Architect and develop scalable, RESTful APIs using Node.js and Express frameworks to serve data to client applications. 
  • CLO2: Implement persistent data storage solutions using a relational database (e.g., MySQL or MariaDB) to manage application data structures. 
  • CLO5: Apply authentication and authorization security best practices to protect user data and API endpoints. 
  • CLO6: Deploy full-stack applications to modern cloud hosting platforms using continuous integration/continuous deployment (CI/CD) workflows.

Directions

  1. Project Setup and Architecture
    • Initialize a new Git repository. Do not use a "monorepo" unless you are comfortable with advanced deployment; keeping the API in its own repo is recommended for simplicity.
    • Connect your Node.js application to your SQL database (using an ORM like Sequelize/TypeORM or a driver like mysql2). 
    • Implement the 3 Entities defined in your Project Proposal:
      • Users Table: Must handle secure password storage (hashing with bcrypt or similar).
      • Resource A: Your main data entity (e.g., Products, Tickets).
      • Resource B: Your related data entity (e.g., Reviews, Comments).
    • Requirement: You must define and enforce the relationships (One-to-Many or Many-to-Many) between these tables using Foreign Keys.
  2. Authentication and Security
    • Implement JSON Web Token (JWT) authentication.
    • Create public routes for POST /register and POST /login.
    • Create middleware to protect private routes (verify the token before allowing access).
    • Requirement: Passwords must never be stored in plain text.
  3. API Endpoints (CRUD Functionality)
    • Build standard RESTful routes for your resources. At a minimum, your API must support:
      • GET (Read all & Read single by ID)
      • POST (Create new)
      • PUT (Update existing)
      • DELETE (Remove)
    • Requirement: Ensure proper HTTP status codes are returned (e.g., 200 for OK, 201 for Created, 401 for Unauthorized, 404 for Not Found, 500 for Server Error).
  4. Documentation
    • Since there is no Frontend yet, you must prove your API works using documentation.
    • Create a Postman/Talend Collection (or Swagger/OpenAPI docs) that includes example requests for every endpoint.
    • Export this collection as a JSON file and include it in your repository, OR include it as a separate document to the Individual Project Phase 1 submission folder. 
  5. Deployment
    • Deploy your API to a public cloud provider (e.g., Render, Versel, or Heroku).
    • Your database must also be hosted (e.g., using the hosting provider's managed database service or a cloud DB like Aiven or PlanetScale).
    • Verification: You must provide a live URL (e.g., https://my-api.onrender.com/) that responds to requests.
